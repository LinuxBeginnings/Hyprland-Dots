# Waybar Hyprland Workspace Click Issue Notes

4/27/26 filed issue with Waybar GH team 
https://github.com/Alexays/Waybar/issues/5008


## Summary
Clicking numbered workspace buttons in `hyprland/workspaces` does not switch to the clicked workspace. The active config uses `hyprland/workspaces#rw` with persistent workspaces and window rewrite output.
## Test performed
Removed this line from the active `hyprland/workspaces#rw` config:
```jsonc
"on-click": "activate",
```
This was removed from both:
- `config/waybar/ModulesWorkspaces`
- `/home/dwilliams/.config/waybar/ModulesWorkspaces`
The Waybar source checkout under `/home/dwilliams/Projects/LinuxBeginnings/Waybar` was not modified.
After fully killing and restarting Waybar, the visual symptom changed:
- Before removing `on-click`, clicking a workspace number made the number disappear and left only an outline.
- After removing `on-click`, clicking a workspace number makes that workspace indicator light up as selected, but Hyprland does not actually move to that workspace.
This suggests the generic `on-click` line was causing a UI/event side effect, but it was not the root cause of the workspace-switch failure.
## Manual dispatcher tests
The old-style dispatcher used by Waybar's Hyprland workspace click handler fails under the current Hyprland Lua dispatch behavior:
```sh
hyprctl dispatch workspace 2
```
Result:
```text
[string "hl.dispatch(workspace 2)"]:1: ')' expected near '2'

 → Note: dispatch in lua is a shorthand for hl.dispatch(...), your syntax might need to be updated.
```
The raw-dispatch wrapper used in some config scroll commands returned `ok` but did not change the active workspace:
```sh
hyprctl dispatch 'hl.dsp.exec_raw("workspace 2")'
```
Observed active workspace stayed on `1`.
The Lua focus dispatcher does switch workspaces correctly:
```sh
hyprctl dispatch 'hl.dsp.focus({ workspace = 2 })'
```
Observed active workspace changed from `1 1` to `2 2`, then was restored with:
```sh
hyprctl dispatch 'hl.dsp.focus({ workspace = 1 })'
```
## Why this test matters
`on-click` belongs to Waybar's generic module event handling. In `src/AModule.cpp`, generic `on-click` values are treated as shell commands to execute. For `hyprland/workspaces`, `"activate"` does not appear to be a documented module-specific action.
The Hyprland workspace module already has its own per-workspace click handler in `src/modules/hyprland/workspace.cpp`.

## Expected behavior
Clicking workspace `2`, `3`, etc. should switch to that workspace.
For normal numeric workspaces, Waybar's Hyprland module source appears to intend to dispatch:
```text
dispatch workspace <id>
```
or, when `move-to-monitor` is enabled:
```text
dispatch focusworkspaceoncurrentmonitor <id>
```
## Actual behavior
Clicking a workspace indicator does not switch to the selected workspace.
## Relevant Waybar source areas
### `src/modules/hyprland/workspace.cpp`
`Workspace::Workspace(...)` connects each workspace button to `Workspace::handleClicked`.
`Workspace::handleClicked(...)` dispatches:
- normal numeric workspace: `dispatch workspace <id>`
- with `move-to-monitor`: `dispatch focusworkspaceoncurrentmonitor <id>`
- named workspace: `dispatch workspace name:<name>`
- special workspace: `dispatch togglespecialworkspace ...`
On the tested Hyprland Lua build, the normal numeric form fails from `hyprctl` with a Lua parse error. The working equivalent for numeric workspace focus is:
```text
dispatch hl.dsp.focus({ workspace = <id> })
```
### `src/modules/hyprland/workspaces.cpp`
`Workspaces::Workspaces(...)` calls:
```cpp
AModule(config, "workspaces", id, false, false)
```
`Workspaces::handleScroll(...)` also sends old-style workspace dispatches:
```text
dispatch workspace e+1
dispatch workspace e-1
dispatch workspace m+1
dispatch workspace m-1
```
### `src/AModule.cpp`
If a module config contains generic event fields like `on-click`, `AModule` connects its own button press handler. That generic handler treats the value as a command string, not as "activate the clicked Hyprland workspace".

## Issue body draft
### Summary
Clicking a numbered workspace in `hyprland/workspaces` does not switch to that workspace. The module displays the workspaces correctly, including persistent workspaces, but mouse clicks on workspace numbers/icons do not move focus to the clicked workspace.
### Environment
- Waybar: `v0.15.0`
- Hyprland: `0.54.0`
- Hyprland commit: `9e357f2481fedfa75899d9721fe1cfc581b3bfc0`
- Waybar source checked against commit: `cca8dc38`
- Module: `hyprland/workspaces`
### Relevant config
See the `hyprland/workspaces#rw` config above.
### Expected behavior
Clicking workspace `2`, `3`, etc. should switch to that workspace.
### Actual behavior
Clicking a workspace indicator does not switch to the selected workspace.
### Source-code area that looks relevant
In `src/modules/hyprland/workspace.cpp`, each `Workspace` creates a `Gtk::Button` and connects it to `Workspace::handleClicked`. That handler dispatches the workspace switch directly through Hyprland IPC.
The root issue appears to be that Waybar sends old-style Hyprland dispatcher strings:
```text
dispatch workspace <id>
```
but this Hyprland build parses dispatch arguments as Lua:
```text
hl.dispatch(workspace 2)
```
which fails with a Lua parse error.
The working manual command is:
```sh
hyprctl dispatch 'hl.dsp.focus({ workspace = 2 })'
```
The removed generic `"on-click": "activate"` line caused a separate visual/event side effect, but removing it did not restore workspace switching.
### Reproduction steps
1. Use `hyprland/workspaces` with persistent workspaces.
2. Include generic `"on-click": "activate"` in the module config.
3. Start Waybar.
4. Click a numbered workspace button, e.g. workspace `2`.
5. Observe that the workspace does not change.
### Control tests
Remove:
```jsonc
"on-click": "activate",
```
Restart or reload Waybar and test workspace clicks again.
Result: visual behavior changed, but workspace switching still did not work.
Manual dispatcher test:
```sh
hyprctl dispatch workspace 2
```
Result: Lua parse error.
Manual Lua focus test:
```sh
hyprctl dispatch 'hl.dsp.focus({ workspace = 2 })'
```
Result: active workspace changed successfully.
